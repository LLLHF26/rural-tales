## 一、项目背景与目标

### 1.1 项目背景

当前乡村旅游普遍存在以下痛点：

- **体验同质化严重**：多数乡村旅游停留在"农家乐+采摘+观光"的浅层模式。
- **文化挖掘不足**：乡村的历史故事、非遗技艺、民俗传说未能有效转化为可感知的旅游体验。
- **缺乏深度互动**：游客以被动游览为主，参与感弱，复游率低。
- **难以吸引年轻群体**：传统乡村游对 Z 世代缺乏吸引力，他们追求沉浸感、互动性的消费体验。
- **剧本策划门槛高**：中小型乡村缺乏专业的剧本策划能力和数字化工具。

### 1.2 项目目标

构建一套 **"AI 剧本生成 + 沉浸式互动导览"** 系统，核心目标：

1. **AI 驱动内容生产**：根据乡村的历史故事、非遗文化、自然景观，自动生成多线分支的沉浸式互动剧本。
2. **沉浸式互动体验**：游客通过手机网页与 AI 虚拟角色对话，完成解谜、探索等任务，实现"人在剧中游"。
3. **LBS + AR 虚实融合**：利用地理位置服务将剧情节点与真实场景绑定；通过图像识别 AR 将虚拟角色、道具、线索叠加到现实场景中，实现"扫实景、见剧情"。
4. **数据化运营**：为管理端提供剧本管理、游客数据查看等基础运营能力。

### 1.3 目标用户

| 用户类型 | 描述 |
|----------|------|
| **游客（C端）** | 年轻游客群体，追求新颖、互动化的旅游体验 |
| **管理员（B端）** | 乡村文旅项目运营者，负责剧本创作、内容管理 |

### 3.2 核心流程

#### 游客完整体验流程

```
打开手机网页
  → 注册/登录
  → 浏览剧本列表 → 选择剧本 → 查看详情 → 领取剧本
  → 到达乡村 → GPS 签到确认到达
  → 剧本开场（AI NPC 对话引入故事背景）
  → 接收第一个任务 → LBS 导航到任务点
  → 到达触发区域 → 自动触发剧情（NPC 对话 / 任务弹出）
  → 完成任务（解谜/拍照/答题/签到）→ 获得线索/道具
  → NPC 对话引导下一节点 → ...（循环推进）
  → 达成结局 → 评分
```

#### AI NPC 对话流程

```
游客点击 NPC / 到达触发点
  → 后端拉取：当前剧情节点的上下文 + 该 NPC 角色设定
  → 组装 System Prompt + 历史对话 → 发送至大模型 API
  → AI 生成符合角色人设的回复（含线索暗示 / 任务指引）
  → 前端流式展示回复 → 游客继续对话
  → 对话记录保存 → 更新剧情进度
```

**对话约束：**
- 角色人设 Prompt：定义性格、语言风格、知识边界。
- 剧情边界控制：AI 不可透露未解锁的剧情，不可偏离主线。
- 内容安全过滤：敏感词过滤。

### 3.3 任务类型

| 任务类型 | 描述 | 触发方式 |
|----------|------|----------|
| **位置签到** | 到达指定 GPS 坐标即完成 | GPS 围栏自动检测 |
| **解谜/答题** | 根据 NPC 对话线索推断答案或回答文化知识题 | NPC 对话中触发 |
| **拍照打卡** | 按要求拍摄指定场景 | 到达位置后 NPC 指引 |
| **选择决策** | 在剧情分支点做出选择，影响后续剧情走向 | NPC 对话中弹出选项 |
| **AR 扫描** | 扫描村落地标/图案/牌匾，识别后弹出虚拟线索或道具 | 进入 AR 扫描页面触发 |

## 五、AI 能力设计

### 5.1 AI 应用场景

| 场景 | 说明 | 技术方案 |
|------|------|----------|
| **剧本自动生成** | 根据乡村文化素材生成剧本框架 | LLM + Prompt Engineering |
| **NPC 智能对话** | 以角色身份与游客开放域对话 | LLM + System Prompt + 上下文管理 |
| **谜题自动生成** | 根据文化知识点生成谜题 | LLM + 题目模板 |
| **场景图生成** | 根据章节/节点描述生成剧情场景图 | Stable Diffusion / 通义万相 |
| **NPC 立绘生成** | 根据角色设定生成 NPC 人物立绘 | Stable Diffusion / 通义万相 |

### 5.2 AI 图片生成策略

**生成时机**：管理端配置剧本时，管理员填写场景/NPC 描述后，点击"AI 生成图片"，后端调用文生图 API。

**图片类型与 Prompt 模板**：

| 类型 | Prompt 要素 | 建议模型 |
|------|------------|----------|
| **场景图** | 乡村名称 + 地点描述 + "古风/写实/电影感, 16:9, 高清" + 季节/时间 + 负面词(现代元素、文字、水印) | Stable Diffusion XL |
| **NPC 立绘** | 角色名称 + 性别/年龄 + 外貌特征 + 服饰风格 + "古风人物立绘, 半身像, 纯色背景, 游戏原画风格" | Stable Diffusion / 通义万相 |

**生成流程**：
```
管理员填写画面描述 → 后端拼装完整 Prompt（含固定正向/负向词）
  → 调用 AI 图片 API（异步）
  → 返回生成结果 URL → 管理员预览 → 确认使用/重新生成
  → 确认后图片存入 OSS → 更新剧本场景图/NPC头像字段
```

### 5.2 AI 剧本生成 Prompt 策略

```
你是一位沉浸式剧本策划师。请根据以下乡村文化素材，生成一份互动剧本：

【乡村信息】
- 名称：{{villageName}}
- 历史背景：{{history}}
- 非遗文化：{{intangibleHeritage}}
- 自然景观/地点：{{scenery}}
- 民间故事/传说：{{legends}}

【剧本要求】
- 类型：{{scriptType}}（悬疑解谜/历史文化/亲子探险）
- 预计时长：{{duration}} 分钟
- 难度：{{difficulty}}/5

【输出要求】
1. 故事主线（200字以内）
2. 3-5个章节，每章2-4个剧情节点
3. 2条以上剧情分支，至少2个不同结局
4. 每个NPC的角色设定（名称、性格、与剧情的关系）
5. 至少5个互动任务/谜题，融入本地文化元素
6. 每个任务推荐GPS触发位置（与乡村实际地点对应）
```

```
你是{{villageName}}的{{roleName}}（{{roleNickname}}），{{age}}岁，

## 六、LBS 功能设计

H5 端使用浏览器 `navigator.geolocation` API 获取 GPS 坐标，结合腾讯地图/高德地图 JavaScript API 展示地图和路线。

| 功能 | 说明 | 精度要求 |
|------|------|----------|
| **实时定位** | 地图上显示游客当前位置 | ≤10米 |
| **地理围栏触发** | 进入指定区域自动弹出剧情/任务 | 围栏半径 20-50米 |
| **任务点标注** | 地图上展示所有任务点位置与状态 | — |
| **路线引导** | 步行路线指引至下一任务点 | — |
| **打卡签到** | 到达任务点后 GPS 确认到达 | ≤15米 |

注意：H5 端 Geolocation API 需要 HTTPS 环境才能正常工作（与 WebRTC 要求一致）。

## 七、AR 功能设计

### 7.1 技术选型

- **图像识别追踪**：[mind-ar-js](https://github.com/hiukim/mind-ar-js)（开源、纯前端、支持浏览器端图像追踪）
- **3D 内容渲染**：three.js（加载 glTF/glb 模型，渲染虚拟物品和 NPC 形象）
- **相机调用**：浏览器 `getUserMedia` API（调用手机摄像头）

mind-ar-js 原理：使用 TensorFlow.js 在浏览器中运行神经网络模型，对相机帧进行实时特征提取与图像匹配，无需后端服务。

### 7.2 AR 功能清单

| 功能 | 说明 | 交互流程 |
|------|------|----------|
| **AR 地标扫描** | 扫描村落的牌匾、古建筑、壁画等预设识别图，识别成功后叠加剧情信息/虚拟线索 | 打开 AR 扫描 → 对准识别图 → 识别成功 → 弹出虚拟内容（文字/图片/道具） |
| **AR 虚拟物品收集** | 在实景中"发现"隐藏的虚拟道具（如古老的钥匙、失传的族谱残页），收集到背包 | 到达指定 GPS 位置 → NPC 提示此处有线索 → 打开 AR 扫描 → 识别环境 → 虚拟物品浮现 → 点击收集 |
| **AR 合影** | 将 NPC 虚拟形象叠加到相机画面中，游客与 NPC 在真实场景中合影 | NPC 对话中选择"合影" → 打开 AR 相机 → NPC 形象叠加画面 → 调整位置 → 拍照保存 |

### 7.3 技术实现要点

```
mind-ar-js 图像识别 AR 核心流程：
  浏览器 getUserMedia 获取相机流
    → mind-ar-js 加载预编译的识别图特征（.mind 文件）
    → 逐帧进行图像特征匹配
    → 匹配成功 → 返回识别图在画面中的变换矩阵（位置/旋转/缩放）
    → three.js 根据变换矩阵在识别图位置渲染 3D/2D 虚拟内容
    → 支持交互：点击虚拟物品收集、查看线索详情

AR 合影流程（简化 2D 方案）：
  浏览器打开相机 → 画面底部或指定区域显示 NPC 立绘/2D 形象
    → 游客可拖拽调整 NPC 位置和大小
    → 点击拍照 → Canvas 合成相机画面 + NPC 形象 → 下载保存
  可选增强：mind-ar-js 平面检测 + three.js 3D 角色模型渲染
```

**uniapp 中的实现方式**：
- AR 页面使用 uniapp 的 `<camera>` 组件（H5 下映射为 WebRTC getUserMedia）
- mind-ar-js 和 three.js 通过 npm 安装引入
- 识别图特征文件（.mind）和 3D 模型文件放在 OSS 上，运行时动态加载
- 不依赖微信环境，任何支持 WebRTC + WebGL 的浏览器均可运行

### 7.4 管理端 AR 配置

在剧本编辑器中，管理员可为任务节点配置 AR 资源：

- **识别图**：上传需要被扫描识别的图片（如村口牌坊、古井石碑等实景照片）
- **叠加内容**：识别成功后显示的内容（虚拟道具图标、文字线索卡、剧情动画）
- **收集道具**：AR 扫描后可收集的虚拟物品（名称、图标、描述、背包展示图）

### 7.5 降级策略

当浏览器不支持 WebRTC/WebGL 或 AR 识别失败时，提供降级方案：
- AR 扫描 → 降级为"拍照上传"模式（手动拍摄目标物体，后端简单校验或直接弹出线索）
- AR 合影 → 降级为"选择 NPC + 选择背景图"拼图合影
- AR 收集 → 降级为直接点击"搜寻"按钮概率获取道具
- 不支持 WebGL 的极低端设备 → 隐藏 AR 入口，仅展示 LBS + 对话体验

| 指标 | 要求 |

- **移动端 H5**：iOS Safari 14+ / Android Chrome 90+（需支持 WebRTC + WebGL）
- **管理端**：Chrome、Edge 最新版（分辨率 ≥1366px）

## 九、数据库核心模型

```
User（用户）
  ├── id, phone, password, nickname, avatar, createdAt

Village（乡村）
  ├── id, name, description, coverImage, location(lat,lng),
  │   address, tags[], createdAt
  ├── Spot[]（打卡点：name, lat, lng, description, images[]）
  └── Culture[]（文化条目：type[history/intangible/legend], title, content）

  │   estimatedDuration, status[draft/published/offline]
  ├── Chapter[] → Node[]（剧情树）
  ├── NPC[]（角色定义 + systemPrompt）
  └── Ending[]（结局定义 + 达成条件）

ScriptProgress（剧本进度）
  ├── id, userId, scriptId, status[playing/completed]
  ├── currentNodeId, completedNodeIds[], unlockedClues[],
  │   items[], npcRelationMap{}
  └── startedAt, completedAt

Task（任务）
  ├── id, nodeId, type[gps_checkin/puzzle/photo/choice/ar_scan],
  │   title, description, answer, rewardItem
  ├── triggerGps { lat, lng, radius }

ChatLog（对话记录）
  ├── id, userId, scriptId, npcId, nodeId
  ├── role[user/npc], content, createdAt

ARResource（AR资源）
  ├── id, scriptId, nodeId, type[recognition_image/collectable/npc_model]

## 十一、风险与应对

| 风险 | 应对 |
|------|------|
| **AI 生成质量不稳定** | 管理端加入人工审核编辑环节；持续优化 Prompt |
| **AI 对话 API 延迟高** | 前端做流式输出体验；设置超时降级回复 |
| **乡村网络信号差** | 核心页面做预加载；关键数据本地缓存 |
| **GPS 定位偏差** | 适当扩大触发半径（20-50m）；支持手动确认到达 |
| **AI 内容安全合规** | 接入敏感词过滤；Prompt 中设置内容边界约束 |
| **AR 设备/浏览器兼容性差** | 低端机型/不支持 WebGL 时走降级方案；需要 HTTPS 部署（WebRTC 强制要求） |
| **AR 识别率低** | 选择特征明显的识别图（牌匾、地标建筑）；mind-ar-js 支持调节匹配阈值；手动重试 |

## 十二、验收标准

1. 移动端可完成"选择剧本 → NPC 对话 → 完成任务 → 达成结局"的完整闭环。
2. AI 可生成含 3 个章节、2 条分支线的基础剧本框架。
3. AI NPC 对话延迟 ≤5 秒，回复符合角色设定且不偏离剧情主线。
4. LBS 定位可在 30 米范围内触发剧情节点的自动推进。
5. AR 扫描可在 2 秒内完成图像识别并弹出对应虚拟内容。
6. AR 合影可将 NPC 形象叠加到实景画面中并保存照片。
7. 管理端可完成剧本的可视化编辑（增删节点、配置 NPC 与任务、上传 AR 识别图）。
8. 管理端可查看基础数据统计（体验人次、完成率、评分）。
9. 所有列表页支持分页与搜索。
10. 系统支持 50 人同时在线进行剧本体验。
11. 通过基础安全测试（SQL 注入、XSS、越权访问）。
12. AI 对话内容安全过滤覆盖。
