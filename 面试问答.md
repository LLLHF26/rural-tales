# 沉浸式乡村文旅剧本创作与智能导览系统 —— 面试问答

## 一、项目整体

### Q1: 介绍一下这个项目的核心业务和技术架构？

**业务背景**：面向乡村文旅场景，游客进入景区后，系统自动分配一个互动剧本，通过地图导航前往各个任务节点，途中可以与 AI NPC 对话、用 AR 扫描收集道具，完成剧本后获得纪念品。

**架构**：前后端分离 + 移动端 H5。

```
移动端 (uni-app/Vue3/H5)
      │
      ▼
FastAPI 后端 ─── MySQL 8.0 (主库)
  │   │   │
  │   │   └── Redis (缓存/验证码)
  │   │
  │   ├── DeepSeek API (剧本生成 + NPC 对话)
  │   ├── 智谱 GLM API (AI 图片生成)
  │   ├── 阿里云 OSS (图片/模型存储)
  │   └── 高德地图 API (导航/POI)
  │
  ▼
管理后台 (Vue3/Element Plus)
```

**为什么选 FastAPI**：
- 原生异步（`async/await`），适合 I/O 密集型场景（调用 3 个外部 AI API）
- 自动生成 OpenAPI 文档，方便前后端联调
- Pydantic 类型校验，减少运行时错误

**为什么 H5 而不是原生小程序**：
- AR 功能需要 WebRTC + Canvas API，小程序限制多
- 一套代码可部署到多个平台

## 二、数据库设计

### Q3: 数据库有哪些核心表？关系是怎样的？

核心表 14 张，按职责分为 5 组：

```
Script（剧本）
  ├── ScriptTask（剧本含多个任务节点）
  │     ├── TaskNode（节点：NPC对话/NPC形象/剧情/结局）
  │     └── ARResource（节点的 AR 资源，含 ArUco 标记 ID 和 3D 模型 URL）
  │
ScriptProgress（用户游玩进度，关联 Script + User + 当前 Node）
  └── ItemCollect（用户在哪个节点收集了什么道具）
```

关系：
- `Script` 1:N `ScriptTask` — 一个剧本多个任务
- `ScriptTask` 1:N `TaskNode` — 一个任务多个节点
- `TaskNode` 1:1 `ARResource` — 一个节点可选关联 AR 资源
- `ScriptProgress` 记录用户在哪个剧本的哪个节点

**分支剧情怎么实现？**
`TaskNode` 表有 `next_node_id` 和 `branch_condition` 字段。NPC 对话节点根据用户选择，返回不同的 `next_node_id`，实现剧情分叉。

### Q4: 数据库连接池怎么配置的？并发怎么处理？

```python
# 使用 SQLAlchemy 2.0 异步引擎
engine = create_async_engine(
    settings.db_url,
    pool_size=20,        # 常驻连接
    max_overflow=40,     # 峰值额外连接（最多 60）
    pool_recycle=3600,   # 1小时回收连接
    pool_pre_ping=True,  # 使用前检测连接有效性
)
```

**关键设计**：
- 每个请求通过 `async def get_db()` 依赖注入获取独立 session
- `pool_pre_ping=True` 防止 MySQL 8小时断连导致的 `MySQL has gone away`
- AI 调用是长耗时操作，用 `async with db.begin()` 事务包裹，失败自动回滚

## 三、AI 服务集成

### Q5: 项目接入了哪些 AI 服务？如何统一调用的？

三个 AI 服务：

| 场景 | 模型 | 作用 |
|------|------|------|
| 剧本生成 | DeepSeek v4-pro | 根据文化主题生成多分支剧本 |
| NPC 对话 | DeepSeek v4-flash | 角色扮演对话 + 上下文记忆 |
| 图片生成 | 智谱 CogView-4 | 生成 NPC 立绘、场景图 |

**统一调用**：抽象了 `BaseLLMProvider`，不同服务实现各自子类：

```python
class BaseLLMProvider(ABC):
    @abstractmethod
    async def chat(self, messages: list[dict], **kwargs) -> str:
        ...

class DeepSeekProvider(BaseLLMProvider):
    async def chat(self, messages, **kwargs):
        async with httpx.AsyncClient(timeout=120) as client:
            resp = await client.post(
                f"{self.base_url}/chat/completions",
                headers={"Authorization": f"Bearer {self.api_key}"},
                json={"model": self.model, "messages": messages, ...}
            )
            return resp.json()["choices"][0]["message"]["content"]
```

**超时与重试策略**：`httpx` 设置了 120 秒超时，配合 3 次指数退避重试（1s → 2s → 4s）。

### Q6: NPC 对话的上下文记忆是怎么实现的？

NPC 对话需要记忆历史，但 DeepSeek API 是无状态的。方案：

```python
# 用 Redis 存储对话历史
key = f"npc_chat:{progress_id}:{node_id}"
history = await redis.get(key)  # JSON 数组

# 拼接 system prompt + 历史 + 新消息
messages = [
    {"role": "system", "content": npc_persona},  # NPC 人设
    *history[-20:],                               # 最近 20 轮
    {"role": "user", "content": user_input},
]
```

**关键设计**：
- 只保留最近 20 轮（`CHAT_CONTEXT_ROUNDS=20`），防止 token 超限
- 每次回复后更新 Redis，带 2 小时 TTL（和剧本游玩时长匹配）
- System prompt 动态注入 NPC 人设（姓名、性格、背景故事）

### Q7: 剧本生成是怎么做的？如何保证剧本质量？

**输入**：管理员在后台填写文化主题、背景、风格偏好（推理/情感/喜剧等）

**Prompt 工程**：
```
你是一位剧本创作大师。根据以下信息创作一个互动剧本：
- 主题：{theme}
- 背景：{background}
- 风格：{style}
- 任务数量：{task_count}

要求：
1. 每个任务包含：标题、场景描述、NPC 对话选项
2. 至少 2 个分支剧情点
3. 输出 JSON 格式：{"tasks": [{"title": "", "nodes": [...]}]}
```

**质量控制**：
- `temperature=0.8` 保证创造性
- `response_format={"type": "json_object"}` 约束输出结构
- 后端解析 JSON 后校验必填字段，失败时重试

## 四、AR 模块

### Q8: AR 扫描的技术实现是怎样的？

**技术选型**：A-Frame 1.2.0 + AR.js，在 `<a-scene>` 中内嵌 `<a-marker>` 和 `<a-entity>` 实现 3D 叠加。

**流程**：
```
加载 A-Frame/AR.js → 创建场景 → 挂载相机 → 检测 ArUco 标记 → 渲染 3D 模型
```

**标记类型**：ArUco barcode 标记（`type="barcode"`），AR.js 内置支持，无需训练 .patt 文件。

```html
<a-scene arjs="sourceType: webcam; detectionMode: mono_and_matrix; matrixCodeType: 3x3;">
  <a-marker type="barcode" value="5">
    <a-entity gltf-model="url(robot.glb)" scale="0.5 0.5 0.5"></a-entity>
  </a-marker>
  <a-entity camera></a-entity>
</a-scene>
```

### Q9: AR 模块遇到过哪些坑？怎么解决的？

**坑 1：模型不显示**
- 现象：标记检测到，但 3D 模型不出现
- 原因：`.glb` 文件的 MIME type 返回 `text/plain`，浏览器不识别为 3D 模型
- 解决：在 FastAPI 启动时注册 `mimetypes.add_type("model/gltf-binary", ".glb")`

**坑 2：返回页面后布局变形**
- 原因：A-Frame 在 `document.body` 和 `document.documentElement` 注入内联样式（`position:fixed; overflow:hidden; width:100%; height:100%`）和类名（`.a-body`），同时注入多个 `<style>` 标签
- 解决：多层清理策略
  1. `destroyAR()` 中完全清空 `body.style.cssText` 和 `html.style.cssText`
  2. 遍历 `document.styleSheets`，删除含 `a-scene`/`a-body` 等选择器的 CSS 规则
  3. 用 `setProperty('overflow', 'auto', 'important')` 覆盖残留样式表规则
  4. `App.vue` 的 `onShow` 生命周期做全局兜底清理

**坑 3：AR 相机无法关闭**
- 原因：AR.js 内部创建了新的 `<video>` 元素，不在我们管理的 `sceneRoot` 中
- 解决：`destroyAR()` 中 `document.querySelectorAll('video')` 遍历所有 video 元素，逐一停止 `srcObject` 的 track

**坑 4：脚本加载时序**
- 现象：`AFRAME is not defined` 或场景 HTML 元素升级失败
- 原因：先用 `innerHTML` 插入 `<a-scene>` 再加载 A-Frame 脚本，浏览器将其当作 `HTMLUnknownElement`
- 解决：必须先 `await loadScript(AFrame)` → `await loadScript(AR.js)` → 再创建并挂载场景 DOM

### Q10: 为什么用 ArUco 标记而不是二维码？

1. **AR.js 原生支持**：`type="barcode"` 直接识别 3x3/4x4 ArUco 矩阵码，无需额外训练
2. **识别速度快**：ArUco 用二进制矩阵，比图像特征匹配（pattern marker）快 3-5 倍
3. **环境适应性强**：对不同光照、角度的鲁棒性更好
4. **可量产**：50 个不同 ID 对应 50 个标记点，系统内含 50 套 .patt 文件 + 对应 .png 标记图

## 五、移动端

### Q11: uni-app 的生命周期在 AR 场景下有什么特殊处理？

AR 场景需要相机权限和 DOM 操作，生命周期管理很关键：

```javascript
onLoad(options) {
    // 检查平台（仅 H5 支持）
    if (typeof document === 'undefined') {
        this.platformError = 'AR功能仅支持浏览器H5环境'
        return
    }
    this._active = true  // 页面活跃标记
    this.startAR()
},
onHide() {
    this._active = false  // 阻止异步回调继续执行
    this.destroyAR()      // 释放相机
},
onUnload() {
    this._active = false
    this.destroyAR()
}
```

**核心思路**：用 `_active` 标记位防止页面隐藏后的异步回调（如 `setTimeout`、API 响应）继续操作已销毁的 DOM。所有异步操作后都检查 `if (!this._active) return`。

### Q12: 页面间的数据通信怎么做的？

AR 扫描页收集道具后，需要通知游玩页更新状态：

```javascript
// AR 页面返回前
notifyPrevPage(result) {
    const pages = getCurrentPages()
    const prev = pages[pages.length - 2]  // uniapp 页面栈
    if (prev) {
        prev.markTaskDoneFromAR(this.taskId, result.item)
        prev.loadNodeAndUpdate()  // 刷新当前节点
    }
}
```

直接在页面栈中调用前一页的方法，无需 Vuex/EventBus，减少全局状态。

## 六、安全与 CORS

### Q13: CORS 配置踩过什么坑？

**错误配置**：
```python
allow_origins=["*"]
allow_credentials=True
```

**问题**：CORS 规范规定，`Access-Control-Allow-Origin: *` 和 `Access-Control-Allow-Credentials: true` 不能同时使用。浏览器会直接拒绝请求。

**修复**：列出具体 origins：
```python
allow_origins=[
    "http://localhost:8080",
    "http://localhost:5173",
    ...
]
```

## 八、开放性问题

### Q16: 如果日活达到 10 万，系统哪些地方会先出问题？

1. **AI API 调用**：DeepSeek 和智谱有 QPS 限制，需要加请求队列 + 限流
2. **数据库**：`ScriptProgress` 高频读写，需要读写分离（主写从读）
3. **Redis 内存**：NPC 对话历史全存 Redis，需要设置内存上限 + LRU 淘汰
4. **图片加载**：OSS 需加 CDN 加速，3D 模型文件 34MB 太大，需要压缩或按需加载
5. **WebSocket**：当前没有实时通信，多用户协作剧本需要引入 WebSocket

### Q17: 你怎么做性能监控和日志？

- **日志**：Python `logging` 模块，按级别输出，生产环境收集到文件 + ELK
- **慢查询**：SQLAlchemy `echo=True` 开发期打印 SQL，生产用 MySQL 慢查询日志
- **APM**：可接入 Prometheus + Grafana 监控 API 响应时间、AI 调用耗时分布

### Q18: 如果重新设计，你会改进哪些地方？

1. **AR 模块**：拆分为独立 Web Worker，避免阻塞主线程
2. **状态管理**：引入 Pinia/Vuex，统一管理游玩进度，替代页面栈直接调用
3. **API 网关**：用 Nginx/Kong 做统一入口、限流、日志
4. **CI/CD**：GitHub Actions 自动跑测试 + 部署
5. **国际化**：当前硬编码中文，应抽取 i18n 文案
6. **离线支持**：用 Service Worker 缓存关键资源，弱网环境下也能玩
